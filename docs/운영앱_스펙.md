# 공주님 클럽 운영앱 — 설계 스펙 (Spec)

- 프로젝트: CLIMAX 서버 토킹바 운영 웹앱
- 문서 버전: 0.1 (초안)
- 관련 문서: `사업계획서.md`, `합법성_및_보호규정.md`

> 본 앱은 예약제 토킹바 운영을 위한 멀티유저 웹앱이다. 예약 → 실제 업무(처리과정 기록) → 거래 → 정산 자동배분까지 하나로 연결한다.

---

## 1. 목표

1. 예약제 운영: 공주님 출근부 기반으로 예약을 잡고 배정한다.
2. 업무 기록: 실제 처리과정(TC 수령·인계, 타임 진행, 2차, 밴, 추천)을 그대로 기록한다.
3. 정산 자동화: 기록된 거래를 지분·추천 규칙대로 자동 분배해 지급표를 만든다.
4. 권한 분리: 각자 자기 역할의 업무만 처리하고, 민감 권한(비고·수정·정산확정)은 사장만 갖는다.

---

## 2. 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 인증·DB·권한 | Supabase (Postgres + Auth + RLS) | 무료 한도, 로그인·행단위 권한(RBAC) 내장 |
| 이미지 저장 | Supabase Storage | 공주님 프로필 사진 저장 |
| 홍보물 합성 | 클라이언트 캔버스(html-to-image/canvas) | 출근부 → 찌라시 PNG 생성 |
| 프론트엔드 | React + Vite (모바일 우선 반응형) | 가볍고 빠른 SPA |
| 배포 | Vercel 또는 Netlify (무료) | 무료 호스팅, URL 공유 |
| 실시간 | Supabase Realtime | 여러 명이 동시에 보는 예약판 동기화 |

> 가입 필요(무료): Supabase, 배포용 Vercel. 진행 시 단계별 안내.

## 3. 사용자 및 권한 (RBAC)

### 역할
- `owner` (사장/시진핑): 전체 권한
- `staff` (운영스탭 = 웨이터·가드 겸직): 예약·손님·밴·거래 입력
- `promoter` (삐끼): 추천 등록, 본인 실적 조회
- `princess` (공주님): 본인 출근·일정·정산 조회

### 권한 매트릭스
| 모듈 | owner | staff | promoter | princess |
|---|---|---|---|---|
| 멤버·권한 설정 | CRUD | — | — | — |
| 영업 세션 | 생성·마감·수정 | 조회 | — | 조회 |
| 출근부 | 전체 | 작성·수정·배정 | — | 본인 출근 체크 |
| 홍보물(찌라시) 생성 | 생성 | 생성 | 생성 | — |
| 손님 관리 | CRUD | 추가·조회·수정 | — | — |
| 밴 관리 | CRUD | 등록·조회 | — | — |
| 예약 관리 | CRUD | 생성·수정·미루기 | — | 본인 일정 조회 |
| 거래 입력(TC/대화료/2차) | CRUD | 입력·조회 | 추천 등록 | — |
| 추천(손님/공주님) | CRUD | 등록 | 등록·본인 실적 | — |

| 비고/메모 | 작성·수정 | 작성 | — | — |

## 4. 데이터 모델 (엔티티)

### 4-1. users (계정)
- id, 이메일, 역할(role), 연결된 member_id, 활성여부

### 4-2. members (멤버 프로필)
- id, 이름(닉), 유형(princess/staff/promoter/owner)
- **profile_photo_url (★ 프로필 사진)**: 공주님은 모두 프로필 사진 지정 가능. Supabase Storage에 저장. 찌라시 생성에 사용.
- **referred_by (★ 공주님 추천 매핑)**: 이 멤버(주로 공주님)를 데려온 내부직원 member_id. 공주님 등록 시 항상 지정. 정산에서 타임당 1만 지급 근거.
- 가입일, 활성여부, 메모

### 4-3. sessions (영업 세션 / 회차)
- id, 날짜, 상태(준비/영업중/마감), 오픈/마감 시각
- 마감 시 정산 스냅샷 생성

### 4-4. attendance (출근부)
- id, session_id, member_id(공주님)
- 출근 예정(planned) / 실제 출근(checked_in 시각) / 퇴근(checked_out)
- **가용 예약 슬롯 수(웨이터 입력)**: 이 공주님이 받을 수 있는 타임 수
- 출근한 공주님만 예약 배정 대상이 된다.

### 4-4b. 홍보물(찌라시) 생성기
- 출근부에서 "찌라시 생성" 클릭 → 현재 출근중인 공주님들을 객체로 모음
- 각 공주님의 프로필 사진 + 닉 + "출근중" 배지를 한 장의 홍보 이미지로 합성(브랜드 네온 톤)
- PNG로 내보내기 → 디스코드 등에 그대로 홍보 배포
- 옵션(추후): 영업 시간/위치/문구 오버레이, 레이아웃 선택

### 4-5. customers (손님)
- id, **닉(게임 닉네임만으로 식별)**, 메모, 누적 방문/이용 이력
- 예약은 전화(인게임 RP)로 받으므로 앱에는 손님 이름(닉)만 입력한다. 별도 연락처/식별번호 없음.
- 추천인 member_id (손님 추천 = 3만 근거, 입장 시 기록)

### 4-6. bans (밴/블랙리스트)
- id, customer_id 또는 식별정보, 사유, 등록자, 등록시각, 해제여부
- 입장/예약 시 자동 경고

### 4-7. reservations (예약)
- id, session_id, customer_id, princess(member_id), 예정 시간/슬롯, 상태
- 상태: 예정 / 진행 / 완료 / 노쇼 / 취소
- 웨이터가 출근부 가용 슬롯에서 배정·수정·미루기

### 4-8. transactions (거래·업무기록) — 모든 이벤트 1행
공통: id, session_id, 시각, 유형(type), 금액, 손님, 담당공주님, 담당직원, 자동분배 결과(JSON), 비고
- **TC**: 5만. 상태 추적 — `tc_received`(웨이터 수령 버튼) → `handed_to_owner`(사장 인계 버튼). 현금 동선 기록.
- **대화료(타임)**: 25만 / 기본 20분. 처리과정 기록 — `타임 시작`/`타임 종료` 타임스탬프. 종료 시 거래 확정.
  - **연장**: 추가 타임 버튼 제공. 단, 해당 공주님의 다음 슬롯에 예약이 차 있으면 "연장 불가" 알럿(가용 여부 미리 표시). 연장 1회 = 대화료 1타임 추가 발생.
- **2차**: 100만. 웨이터가 기록(공주님 동의 체크), 중개 웨이터 지정.
  - **복귀시간 고려**: 2차 시작 + 예상 복귀시간 동안 해당 공주님은 예약 불가로 자동 블로킹(스케줄에 점유 표시). 복귀 후 다시 예약 가능.
- **추천비(손님)**: -3만(풀) / +3만(추천인).
- **추천비(공주님 영입)**: 대화료 타임 발생 시 자동 -1만(풀) / +1만(referred_by).
- (팁: 기록하지 않음. 받은 사람 개인.)

### 4-9. settlements (정산)
- id, session_id, 생성시각, 상태(임시/확정)
- 라인: member_id, 항목별 금액(대화료/2차/지분/추천/합계), 지급여부
- 사장만 확정·수정 가능

## 5. 정산 엔진 로직 (핵심)

세션 단위로 계산한다. 모든 금액은 게임 내 화폐.

### 5-1. 운영풀(Operating Pool) 집계
```
운영풀 입금:
  + TC 5만        × (TC 건수)
  + 대화료 풀몫 10만 × (대화료 타임 수)
  + 2차 풀몫 25만   × (2차 건수)
운영풀 출금(추천비 차감):
  − 손님 추천 3만   × (추천받은 손님 수)
  − 공주님 추천 1만 × (referred_by가 있는 공주님의 대화료 타임 수)
= 최종 운영풀
```

### 5-2. 지분 분배
```
총지분 = 1.2(사장) + 1.0 × (해당 세션 출근한 운영스탭 수)
각자 지분몫 = 최종 운영풀 × (본인 지분 / 총지분)
```
> 출근부 기준으로 그날 일한 운영스탭만 지분을 받는다.

### 5-3. 개인 정산
- 공주님 i = 대화료 15만 × (본인 타임) + 2차 70만 × (본인 2차 건)
- 운영스탭 j = 지분몫 + 2차 중개 5만 × (본인 접수 2차 건)
- 추천인 k = 손님추천 3만 × (본인 추천 손님) + 공주추천 1만 × (본인이 영입한 공주의 타임)
  - 추천인은 운영스탭일 수도 삐끼일 수도 있음(members.referred_by / customers.추천인 기준)
- 사장 = 지분몫(1.2)
- (팁은 정산 제외, 개인 직접 수령)

### 5-4. 현금 정합성 체크
- TC·대화료·2차 수령액 합계 = 사장 인계 합계 인지 대조(미인계 표시).

## 6. 운영 흐름 (영업 1회)

```
[영업 전] 사장: 세션 생성 → 공주님 출근부 작성(예정)
[오픈]    공주님: 출근 체크 → 웨이터: 가용 슬롯 입력
[예약]    웨이터: 출근한 공주님에 손님 예약 배정/미루기/수정
[입장]    웨이터: 손님 검색/등록(밴 자동확인) → TC 받음 버튼 → 사장 인계 버튼 → 공주님 배정
[진행]    웨이터: 타임 시작 → 종료(대화료 자동 생성) / 2차 기록 / 삐끼·스탭: 추천 등록
[보안]    가드(겸직): 밴 등록
[마감]    사장: 세션 마감 → 정산 자동배분 → 지급표 확정·지급
```

## 7. 화면 (역할별 메인 / 모바일 우선)

- 공통: 로그인, 상단에 현재 세션 상태
- owner: 대시보드(매출·세션·정산), 멤버·권한, 정산 확정·비고
- staff: 오늘 예약판 + 손님 입장 처리(원탭 버튼들) + 타임 시작/종료 + 2차·밴 입력
- promoter: 추천 등록 + 내 실적
- princess: 내 출근 체크 + 내 일정 + 내 정산 내역

UX 원칙: 모바일 우선, 큰 버튼·최소 클릭, 타임은 원탭 시작/종료, 실시간 동기화.

---

## 8. 개발 단계

- **V1 (운영 기본기)**: Supabase 세팅 + 인증·권한 + 멤버 + 세션 + 출근부 + 손님 + 밴 + 예약
- **V2 (돈 + 홍보)**: 거래 입력(TC 동선·타임·2차) + 추천 + 정산 엔진 + 지급표 + 프로필 사진/찌라시 생성기
- **V3 (확장)**: 대시보드·매출통계, 삐끼 실적, 공주님 본인 뷰, 백업/내보내기, 찌라시 레이아웃 옵션

---

## 9. 비기능 요구사항

- 모바일 우선 반응형(FiveM 중 폰 사용 가정)
- 실시간 동기화(여러 명 동시 사용)
- 데이터 백업/내보내기(CSV/JSON)
- 모든 활동은 게임 내 RP·게임 내 화폐 (합법성 규정 준수)

---

## 10. 결정된 사항

- 손님 식별: 게임 닉네임만. (전화 예약은 RP, 앱엔 이름만)
- 타임: 기본 20분 + 연장 버튼. 다음 슬롯 예약 시 연장 불가 알럿.
- 2차: V1/V2에 포함. 복귀 예상시간까지 공주님 예약 자동 블로킹.
- 추천: (1) 삐끼 손님 추천 고정 3만/명, (2) 공주 영입자에게 그 공주 타임당 1만(운영풀에서).

## 10-1. 추후 결정

- 찌라시 레이아웃 옵션·문구 템플릿(V3)
- 2차 복귀시간 기본값/캘린더 뷰 형태
