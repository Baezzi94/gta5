# Black Out 정보부 시스템 — MVP 설계서

작성: 2026-07-13 · 상태: **확정** (브레인스토밍 완료, 구현 플랜 작성 전)
배경·결정 경위: `브레인스토밍.md` 참조

## 1. 개요

CLIMAX 서버 갱단 Black Out의 정보부 운영 웹앱. 조직원의 첩보 제보를 정보부가 검증·분류하고, 보안등급(P0~P3)에 따라 계급별로 열람시키는 정보 파이프라인 + 인물 DB.

- 위치: `gta5/갱활동/app/`
- 스택: React 18 + Vite + Supabase(인증·DB·Storage·RLS) + Vercel — 토킹바(공주님클럽)와 동일 패턴, **단 Supabase/Vercel 프로젝트는 완전 별개** (토킹바 코드·데이터 절대 불간섭)
- 모바일 UI 우선 (인게임 브라우저 접속 대응)
- **대외비**: 앱 어디에도 개발자(진핑) 크레딧·흔적을 남기지 않는다
- **직책 비노출**: 시진핑이 정보부장인 것은 조직 내에서도 비밀. 앱은 사용자 계급·직책을 서로에게 절대 노출하지 않는다 (멤버 목록·프로필에 역할 표시 금지). 계급은 오직 "보이는 탭/데이터의 차이"로만 작동

## 2. 인증·온보딩

1. 디스코드 OAuth 로그인 (Supabase Discord provider)
2. 최초 로그인 시 온보딩: **캐릭터명 + 인게임 전화번호 필수 입력** — 미입력 시 가입 미완료
3. 가입 완료 후 상태 = `pending`(승인 대기): 아무 데이터도 볼 수 없음
4. 관리자(정보부장 진핑 + 보스)가 [조직 관리] 화면에서 승인 + 계급 부여 → 활동 시작

## 3. 계급·보안등급

### 계급 (role)

`boss`(대부) / `vice`(부대표) / `branch_head`(지부장) / `manager`(부장) / `team_lead`(팀장) / `member`(조직원) / `intel_chief`(정보부장) / `intel_agent`(정보부원)

- 재정(차미슬)은 MVP에서 `member` 취급. 재정 모듈 도입 시 재검토
- 멤버 승인·계급 변경 권한: `intel_chief` + `boss`만

### 보안등급 열람 매핑 (누적)

| 등급 | 열람 가능 |
|------|-----------|
| P0 | boss, intel_chief |
| P1 | + vice |
| P2 | + branch_head, manager, intel_agent |
| P3 | 전체 승인 조직원 (team_lead, member 포함) |

- intel_agent는 검증 업무상 P2까지. P1·P0 불가
- 열람 제한은 화면 숨김이 아니라 **Supabase RLS로 DB단 강제**

## 4. 데이터 모델

| 테이블 | 핵심 컬럼 | 비고 |
|--------|-----------|------|
| `profiles` | discord_id, 캐릭터명, 전화번호, role, status(pending/active/제명) | auth.users 1:1 |
| `tips` | 제목, 내용, category, status(접수→검토중→채택/기각), 진위판정, clearance(P0~P3), submitter_id | 제보자 신원은 정보부·보스만 조회 (RLS 또는 뷰로 분리) |
| `tip_photos` | tip_id, storage 경로 | Supabase Storage, 등급 따라 접근 제한 |
| `persons` | **전화번호(유니크, nullable)**, 소속(적대조직/시민/경찰 등), 특이사항, clearance | 인물 자체에도 보안등급 |
| `person_names` | person_id, 이름, 등록일 | 이름 이력 = 개명 추적. 같은 번호에 새 이름 → 같은 인물로 축적 |
| `tip_persons` | tip_id, person_id | 제보↔인물 연결. v2 사회관계망의 기반 |
| `categories` | 이름 | 초기값: 적대조직/경찰/시민/내부/기타. 정보부가 앱에서 추가 가능 |

**인물 식별 원칙**: 전화번호가 불변 ID (이름은 개명 가능). 번호 미상 인물은 번호 없이 등록 허용, 번호 확보 시 정보부가 수동 병합. 번호가 같으면 자동으로 한 인물(이름들은 별명 이력으로 그룹).

## 5. 화면 구성 (계급별 탭 노출)

| 화면 | 접근 | 기능 |
|------|------|------|
| 로그인/온보딩 | 전체 | 디코 로그인, 캐릭터명+전화번호 등록. 조직 마크 외 무표기 |
| 정보 제공 | 승인 조직원 전체 | 제보 작성(사진+글+카테고리), **내 제보 목록+처리상태만** 확인 |
| 정보 열람 | 승인 조직원 전체 | 자기 등급 이하의 채택된 정보 열람 |
| 접수함 | intel_agent, intel_chief, boss | 전체 제보 열람(제보자 신원 포함), 분류, 진위 파악, 등급 제안 |
| 인물 DB | intel_agent(열람·등록), intel_chief(병합·관리) | 인물 카드, 이름 이력, 연결된 제보 |
| 등급 확정 | intel_chief | 등급 최종 확정, 채택/기각 확정 |
| 조직 관리 | intel_chief, boss | 멤버 승인, 계급 부여·변경 |

## 6. 핵심 흐름

제보(조직원) → 접수함 등재 → 정보부 분류·진위 파악·등급 제안 → 정보부장 확정 → 채택 시 해당 등급 이상에게 [정보 열람] 공개, 제보자에겐 상태만 갱신. 기각 시 제보자에게 "기각" 상태 표시(사유 노출 여부는 정보부 재량 메모).

## 7. 에러·보안 처리

- 모든 열람/쓰기 권한은 RLS 정책으로 강제 (클라이언트 우회 불가)
- pending 사용자는 어떤 테이블도 SELECT 불가
- 제보자 신원(submitter_id)은 정보부·보스 외 등급과 무관하게 차단
- Storage 사진도 서명 URL + 등급 검사 거쳐 제공
- 전화번호 유니크 충돌(조직원 가입 시 기존 인물 DB와 중복) → 정보부에 알림, 수동 확인

## 8. 테스트

- Vitest + Testing Library (토킹바와 동일)
- 핵심 검증: 등급별 열람 경계(P0~P3 각 계급 조합), 제보자 신원 차단, pending 차단, 전화번호 그룹핑 로직

## 9. 범위 밖 (v2+)

보고서 작성·발행, 사회관계도(전화번호 기준 그래프), 정보 거래(판매 가능/불가 + 정보비 장부), 수금·재정(차미슬 인터뷰 선행), 자산 대시보드, 정산 계산기(라운드 엔빵), 증거 영상 아카이브, 블랙모터스/세차장/하우스 장부(BM 구체화 대기)
